# 赛博居委会调度中枢 — 敏捷 PRD（MVP 版）

---

## 一、产品概述

### 1.1 产品名称

**赛博居委会调度中枢**

### 1.2 产品愿景（One-liner）

打造一个以 **3D 空间可视化** 为核心、**多模态 AI 驱动** 的社区智能工单调度平台，将居民的情绪宣泄转化为结构化的高效派单，消除基层管理摩擦力。

### 1.3 产品定位

面向 **城市社区基层治理** 场景的 SaaS 化智能调度工具。以"空间即界面"为核心理念，通过 3D 数字孪生 + LLM 多模态识别，将传统社区报修/投诉流程从"列表式工单管理"升级为"空间可视化调度"，降低基层管理者的认知负荷。

### 1.4 核心价值主张

| 角色 | 核心价值 |
|------|----------|
| C 端居民 | 拍照吐槽即可上报，全程追踪不焦虑 |
| B 端物业 / 居委会 | 一眼看全局、AI 帮忙分轻重缓急，空间化调度让工单不再是冰冷的列表 |

## 二、目标用户与核心痛点

### 2.1 用户画像

#### C 端居民（数据触发者 / 问题发现者）

| 维度 | 描述 |
|------|------|
| **典型代表** | 小区住户、租客，年龄跨度大（25-65 岁），手机使用能力参差不齐 |
| **核心痛点** | 报修流程繁琐（打电话→登记→等待→反复催办）；不知道排队进度，产生焦躁情绪；反馈后石沉大海，对物业失去信任 |
| **核心诉求** | 极简上报：拍照 + 语音即可；实时可视化排队进度；获得情绪安抚反馈 |
| **使用场景** | 楼道灯坏了、垃圾堆积、水管漏水、邻居噪音等日常生活问题 |
| **使用设备** | PC 端浏览器 / 移动端浏览器（响应式适配，UI 自动适配屏幕尺寸） |

#### B 端物业 / 居委会（系统核心使用者 / 调度决策者）

| 维度 | 描述 |
|------|------|
| **典型代表** | 物业客服、维修班组长、居委会网格员 |
| **核心痛点** | 琐碎工单太多，难以分清轻重缓急（水管爆了 vs 灯泡闪了混在一起）；缺乏全局空间视角，不知道小区"哪里出了问题集中区"；大量无效/重复上报消耗人力 |
| **核心诉求** | 直观看到小区哪里出了严重问题（空间热力图）；AI 帮忙过滤无效警报并自动分类分级；一键派单，减少沟通链路 |
| **使用场景** | 日常监控大屏、高峰时段（暴雨/台风后）集中调度 |
| **使用设备** | PC 端浏览器 / 移动端浏览器 / 大屏（响应式适配，UI 自动适配屏幕尺寸，移动端侧重快速查看与派单，PC/大屏侧重 3D 调度舱全景操作） |

### 2.2 用户痛点映射矩阵

```
          │ C端居民                │ B端物业
──────────┼────────────────────────┼────────────────────────
上报环节  │ 流程繁琐，不知道找谁    │ 重复上报多，信息不标准
──────────┼────────────────────────┼────────────────────────
等待环节  │ 进度不可见，焦虑        │ 无法区分优先级
──────────┼────────────────────────┼────────────────────────
处理环节  │ 反馈延迟，感受差        │ 缺乏空间视角，调度低效
──────────┼────────────────────────┼────────────────────────
闭环环节  │ 不知道处理结果          │ 工单归档散乱，难统计
```

                                                      │

### 3.2 逐步骤详解

| 步骤 | 执行者 | 动作 | 系统行为 |
|------|--------|------|----------|
| **① 触发** | C 端居民 | 选择门牌号 → 拍照/相册 → 语音吐槽 → 提交 | H5 压缩上传 → CDN 存储 → 创建工单记录 (状态: pending) |
| **② 流转** | AI 系统 | — | 异步调用 LLM 多模态接口 → 解析图片+语音 → 输出 JSON `{category, severity, summary}` → 容错校验 → 写入 DB |
| **③ 响应** | B 端大屏 | — | WebSocket 推送 → 对应楼栋 3D 模型切换材质 → 呼吸灯闪烁 (蓝/黄/红按等级) → 侧边栏工单列表刷新 |
| **④ 处理** | B 端物业 | 点击工单详情 → 指派维修人员 → 确认派单 | 工单状态更新 → 3D 模型恢复常态 → C 端排队进度 -1 |
| **⑤ 闭环** | C 端居民 | 查看排队进度 → 收到完成通知 → 评价 | 工单归档 → 评分数据写入 → 统计面板更新 |

### 4.2 模块 A — 极简上报端（H5 移动端 Web）

#### A1. 楼栋级联选择器
- **功能**：两步选择 — 先选楼栋号（如 1 栋、2 栋...），再选单元/楼层/门牌号
- **交互**：下拉级联 or 搜索联想，支持常用地址缓存
- **技术**：数据来源于 Building 表，API 动态拉取
- **MVP 范围**：仅支持楼栋 + 门牌号两级，不细化到楼层

#### A2. 多媒体上传入口
- **功能**：
  - 拍照按钮（调用原生相机）
  - 从相册选择（最多 3 张）
  - 语音输入按钮（长按录音，松开发送，≤60s）
- **交互**：
  - 上传前压缩（图片 ≤ 800px 宽，WebP 格式）
  - 进度条显示上传状态
  - 语音自动转文字预览（可选确认 or 重新录制）
- **技术**：`<input capture="environment">` 拍照，MediaRecorder API 录音，COS/OSS 直传

#### A3. 实时排队进度追踪卡片
- **功能**：
  - 显示"您前面还有 X 人在排队"
  - 状态流转：已提交 → AI 识别中 → 待派单 → 处理中 → 已完成
  - 处理完成后推送评价入口（笑脸/哭脸 + 选填评语）
- **交互**：卡片式设计，一眼可读
- **技术**：轮询 or WebSocket 订阅工单状态变更

---

### 4.3 模块 B — 3D 空间调度舱（PC 桌面端 Web）

#### B1. 等距视角小区 3D 地图
- **视觉风格**：等距/2.5D 视角（参考《城市天际线》简洁风格），非 FPS 自由视角
- **内容要素**：
  - 楼栋方块（按实际小区数据搭建）
  - 小区道路、绿化示意
  - 楼栋顶部标注楼号
- **交互**：
  - 鼠标悬浮：显示楼栋名称 + 当前工单数
  - 鼠标点击：筛选该楼栋的工单列表
  - 滚轮缩放、拖拽平移
- **技术选型**：**Three.js**（WebGL），渲染等距正交相机视角
- **MVP 范围**：硬编码 3-6 栋楼的示意小区模型，不做真实 GIS 接入

#### B2. 动态楼栋发光材质系统
- **状态机定义**：

| 警报等级 | 颜色 | 动画效果 | 触发条件 |
|----------|------|----------|----------|
| 正常 | 灰白色 | 无 | 该楼无待处理工单 |
| 轻度 | 蓝色呼吸灯 | 亮度 0.3→0.7 正弦波，周期 3s | 存在优先级"低"的工单 |
| 中度 | 黄色呼吸灯 | 亮度 0.4→0.9 正弦波，周期 2s | 存在优先级"中"的工单 |
| 重度 | 红色呼吸灯 | 亮度 0.5→1.0 正弦波，周期 1s + 轻微缩放 | 存在优先级"高"的工单 |

- **优先级规则**：一栋楼有多工单时，取最高等级展示
- **技术**：Three.js ShaderMaterial 实现 emissive 呼吸效果，GSAP 驱动动画

#### B3. 实时滚动待办工单侧边栏
- **布局**：左侧 3D 大屏占 70%，右侧侧边栏占 30%
- **内容**：
  - 工单卡片：楼栋号、问题摘要（AI 提取的前 30 字）、紧急度标签、上报时间
  - 支持按等级筛选（全部 / 高 / 中 / 低）
  - 支持按楼栋筛选
  - 新工单进入时顶部滑入动画
- **技术**：虚拟滚动（react-window / vue-virtual-scroller）保证大量工单时性能

#### B4. 工单派发操作台
- **功能**：
  - 点击工单卡片 → 弹出详情面板（原始图片、语音播放、AI 分类摘要）
  - 指派维修人员下拉框
  - "派单"按钮 → 确认 → 状态变更
- **交互**：派单成功后，3D 楼栋灯效即时更新，C 端进度同步

---

### 4.4 模块 C — 神经中枢（Serverless API）

#### C1. 多模态 AI 网关
- **输入**：图片 URL + 语音文件 URL（或语音转文字后的文本）
- **处理流程**：
  1. 图片压缩 / 语音转文字预处理
  2. 组装 Prompt：`"你是一个社区工单分类助手。根据以下图片和居民描述，输出 JSON 格式的事件分类和紧急程度..."`
  3. 调用多模态 LLM（GPT-4o / Claude / 通义千问 VL）
  4. 接收返回的 JSON
- **Prompt 设计要点**：
  - 分类枚举：水电维修 / 环境卫生 / 噪音扰民 / 公共设施 / 安全隐患 / 其他
  - 紧急度标准：低（非紧急，可排队）/ 中（影响生活但不危险）/ 高（安全隐患或影响多人）
  - 输出约束：强制 JSON 输出，包含 `category`、`severity`、`summary`（≤50 字）

#### C2. JSON 解析与容错层
- **容错策略**：
  - LLM 返回非 JSON → 正则提取 JSON 片段 → 重试 1 次
  - JSON 缺少字段 → 默认值填充（category: "其他"，severity: "中"）
  - LLM 超时 → 降级为人工标注队列
- **输出标准化**：统一转换为内部 Schema 再写入 DB

#### C3. WebSocket 实时推送服务
- **推送事件**：
  - `ticket:new` — 新工单创建（AI 分析完成后）
  - `ticket:update` — 工单状态变更（派单/完成）
  - `building:alert_change` — 楼栋警报等级变化
- **连接管理**：心跳保活，断线重连，按小区 ID 划分房间

## 五、核心数据实体（Database Schema 设计）

### 5.1 Building（建筑实体）

```sql
CREATE TABLE buildings (
  id            VARCHAR(36) PRIMARY KEY,          -- UUID
  name          VARCHAR(64)  NOT NULL,            -- 楼栋名称："1栋"、"2栋"
  community_id  VARCHAR(36) NOT NULL,             -- 所属小区 ID
  position_x    FLOAT NOT NULL,                   -- 3D 场景 X 坐标
  position_z    FLOAT NOT NULL,                   -- 3D 场景 Z 坐标 (Three.js 用 Y 轴做高度)
  width         FLOAT DEFAULT 1.0,                -- 楼栋宽度（3D 模型缩放）
  depth         FLOAT DEFAULT 1.0,                -- 楼栋深度
  height        FLOAT DEFAULT 3.0,                -- 楼栋高度（可映射真实层数）
  alert_level   ENUM('normal','low','medium','high') DEFAULT 'normal',  -- 当前最高警报状态（冗余字段，加速查询）
  created_at    TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### 5.2 Ticket（工单实体）

```sql
CREATE TABLE tickets (
  id            VARCHAR(36) PRIMARY KEY,          -- UUID
  building_id   VARCHAR(36) NOT NULL REFERENCES buildings(id),
  reporter_id   VARCHAR(36),                      -- 上报居民 ID（匿名也可提交）
  door_number   VARCHAR(16),                      -- 门牌号：如"101"
  category      ENUM('water_electric','environment','noise','facility','safety','other'),
  severity      ENUM('low','medium','high'),
  ai_summary    VARCHAR(128),                     -- AI 生成的摘要（≤50字）
  original_text TEXT,                             -- 语音转文字后的原始文本
  image_urls    JSON,                             -- ["https://cdn.xxx/img1.webp", ...]
  audio_url     VARCHAR(512),                     -- 语音文件 CDN URL
  status        ENUM('pending','ai_processing','waiting_dispatch','in_progress','completed','cancelled'),
  assignee_id   VARCHAR(36),                      -- 指派的维修人员 ID
  completed_at  TIMESTAMP,
  rating        TINYINT,                          -- 居民评分 1-5
  created_at    TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at    TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  
  INDEX idx_building_status (building_id, status),
  INDEX idx_severity (severity),
  INDEX idx_created (created_at)
);
```

### 6.2 技术选型建议

| 层次 | 推荐方案 | 备选方案 | 选型理由 |
|------|----------|----------|----------|
| H5 前端 | React + Vant UI | Vue + Vant | 生态成熟，移动端组件库丰富 |
| PC 3D 前端 | React + Three.js + React Three Fiber | Vue + TresJS | R3F 生态最活跃，社区资源最多 |
| API 服务 | Next.js API Routes | Express + Serverless | 前后端同仓库，部署简单 |
| 数据库 | PostgreSQL | MySQL | JSON 字段支持更好（image_urls） |
| 实时推送 | Supabase Realtime | Pusher / Socket.io | 自带 DB，MVP 成本极低 |
| LLM | GPT-4o / Claude 3.5 | 通义千问 VL | 多模态能力成熟，JSON 输出稳定 |
| 语音识别 | Whisper API | 阿里云语音 | 准确率高，中文支持好 |
| 文件存储 | Cloudflare R2 | 阿里 OSS | 免流量费，MVP 成本低 |

## 八、非功能性需求

### 8.1 性能指标

| 指标 | 目标值 | 衡量方式 |
|------|--------|----------|
| H5 首屏加载 | ≤ 2s (4G 网络) | Lighthouse |
| 3D 场景加载 | ≤ 3s (桌面端) | Performance API |
| AI 分析延迟 | ≤ 8s (P95) | 服务端埋点 |
| WebSocket 推送延迟 | ≤ 500ms | 端到端时间戳差值 |
| 3D 场景帧率 | ≥ 30 FPS | requestAnimationFrame |

### 8.2 兼容性

| 端 | 最低要求 |
|----|----------|
| H5 上报端 | iOS 14+ Safari / Android 8+ Chrome / 微信内置浏览器 |
| PC 调度舱 | Chrome 90+, Edge 90+（需 WebGL 1.0+） |

### 8.3 安全要求

- 图片上传做格式校验 + 大小限制（≤10MB）
- API 接口做速率限制（上报接口 1次/10s/用户）
- LLM Prompt 注入防护（过滤用户输入中的指令类文本）
- CDN 图片链接加签名，防恶意访问

## 九、成功指标（MVP 验证维度）

### 9.1 北极星指标

**社区工单平均闭环时长**（从居民上报到问题解决的总时长）

### 9.2 关键度量

| 维度 | 指标 | 数据来源 |
|------|------|----------|
| 使用量 | 周活跃居民数 / 周工单量 | DB 统计 |
| 效率 | AI 分类准确率（人工抽查） | 人工抽检 |
| 体验 | 3D 调度舱日活使用时长 | 前端埋点 |
| 满意度 | 居民评价平均分 | Ticket.rating |
| 留存 | B 端周留存率 | 登录日志 |

### 9.3 MVP 验收标准

- [ ] C 端能完成一次完整的"拍照→上报→看到排队进度"流程
- [ ] AI 能正确识别 ≥80% 的工单分类（人工抽检 50 条）
- [ ] B 端 3D 大屏能实时响应新工单（延迟 ≤3s）
- [ ] B 端能完成一次"查看→派单"操作，且 C 端进度同步更新

## 十、风险识别与应对

| 风险项 | 影响 | 概率 | 应对策略 |
|--------|------|------|----------|
| LLM 返回不稳定（JSON 格式错误、幻觉分类） | 高 | 高 | Prompt Engineering 优化 + 多轮容错 + 降级人工队列 |
| 3D 渲染性能在低端设备不达标 | 中 | 中 | 模型面数精简、LOD 策略、降级 2D 版 |
| 用户不上传语音/照片质量差 | 中 | 中 | 强制至少拍照一张，提供示例引导 |
| 物业方培训成本高、使用意愿低 | 高 | 中 | 3D 大屏"好看好玩"，降低学习门槛；先找试点小区 |
| AI 误判紧急度导致安全事故延误 | 极高 | 低 | "高"等级工单强制人工二次确认后派单 |
